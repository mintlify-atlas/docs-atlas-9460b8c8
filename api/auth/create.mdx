---
title: Create Authentication Token
api: POST /v1/admin/authentications

---

## Authentication

Requires a token with `authentications:write` permission.

## Request Body

<ParamField body="project_reference" type="string" required>
  Project reference identifier (max 100 characters).
</ParamField>

<ParamField body="role_id" type="string" required>
  UUID of the role to assign to this token.
</ParamField>

<ParamField body="secret_key" type="string" required>
  The secret token string (max 500 characters). This is the actual Bearer token.
</ParamField>

<ParamField body="email" type="string" required>
  Email address associated with the token (max 255 characters).
</ParamField>

<ParamField body="description" type="string">
  Optional description of the token's purpose (max 500 characters).
</ParamField>

<ParamField body="is_active" type="boolean">
  Whether the token is active. Defaults to `true`.
</ParamField>

<ParamField body="expiration_date" type="string">
  Optional ISO 8601 expiration date. Token becomes invalid after this date.
</ParamField>

  <ResponseField name="project_reference" type="string">

  <ResponseField name="role_id" type="string">

  <ResponseField name="secret_key" type="string">

  </ResponseField>

    Associated email.

  <ResponseField name="description" type="string">
    Token description.
  </ResponseField>
  
  <ResponseField name="is_active" type="boolean">
    Active status.
  </ResponseField>
  
  <ResponseField name="expiration_date" type="string | null">
    Expiration date or null.
  </ResponseField>
  
  <ResponseField name="created_at" type="string">

  </ResponseField>
  
  <ResponseField name="updated_at" type="string">
    ISO 8601 timestamp of last update.
  </ResponseField>
  
  <ResponseField name="removed_at" type="string | null">
    Soft deletion timestamp or null.
  </ResponseField>
</ResponseField>

## Example Request

```bash
curl -X POST https://api.weaponx.io/v1/admin/authentications \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "project_reference": "my-app",
    "role_id": "750e8400-e29b-41d4-a716-446655440002",
    "secret_key": "sk_prod_abc123def456...",
    "email": "developer@company.com",
    "description": "Production API token for backend service",
    "is_active": true,
    "expiration_date": "2026-12-31T23:59:59Z"
  }'
```

## Token Naming Convention

We recommend using a consistent naming convention for tokens:

- `sk_dev_*` - Development tokens
- `sk_staging_*` - Staging tokens
- `sk_prod_*` - Production tokens
- `sk_test_*` - Testing tokens

## Security Best Practices

1. **Generate strong tokens**: Use cryptographically secure random strings
2. **Set expiration dates**: Especially for temporary access
3. **Use descriptive names**: Include purpose and environment in description
4. **Rotate regularly**: Update tokens periodically for production use
5. **Monitor usage**: Track which tokens are actively being used
